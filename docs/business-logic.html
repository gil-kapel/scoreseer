The core engine

A self-improving feedback loop

Most casual prediction has no feedback: you guess, the match happens, you never learn whether your reasoning was sound. ScoreSeer closes that loop. The real "actor" is a scheduler — the human mostly observes.

Before kickoff

1 · Predict

Fetch fresh pre-match context, inject what the system has learned, and commit to a strict structured prediction.

After full-time

2 · Grade

Pull the authoritative result, compare against the prediction, and persist a falsifiable, honest grade.

Rolling history

3 · Calibrate

Recompute observed biases from every grade into a compact snippet that shapes the next round of predictions.

↻  Calibration feeds straight back into step 1 — every graded match makes the next prediction smarter.
Per-fixture lifecycle

Two flows run for every match

Each fixture moves through a prediction flow before kickoff and a grading flow after the whistle. A failure on one fixture never aborts the batch — failures are isolated, logged, and made visible, never silently dropped.

A Prediction flow · before kickoff
  1. Select — the scheduler wakes on a cadence and picks fixtures kicking off inside the prediction window that have no current prediction.
  2. Fetch context — Claude web search gathers form, lineups, injuries, head-to-head, stakes, venue → stored as an immutable DataSnapshot with source citations.
  3. Inject calibration — the current CalibrationProfile (the system's known biases) is folded into the prompt context.
  4. Predict — the LLM returns a schema-validated prediction: score, scorers + likelihoods, confidence, explanation.
  5. Store & show — linked to its snapshot + model/prompt/calibration version, it appears under "Upcoming."
  6. Repair or flag — bad schema → retry with a repair prompt; thin data → low-confidence note; exhausted → a visible "prediction failed" state.
B Grading flow · after full-time
  1. Detect finish — the scheduler notices a predicted match has reached full-time.
  2. Fetch result — the authoritative score & goalscorers come from a structured sports API, saved as a Result.
  3. Grade — compare prediction vs reality: exact hit, outcome, goals error, scorer precision/recall + Brier, confidence Brier.
  4. Calibrate — recompute the CalibrationProfile from rolling grade history; its version increments.
  5. Handle edges — postponed/abandoned → void (excluded from metrics); ET/penalties → exact-score graded on the 90' line, advancing team tracked separately.
The contract

A falsifiable, structured prediction

Generic "predict this match" prompts hallucinate, cite nothing, and never learn. ScoreSeer instead forces the model into a strict schema — validated at the boundary, repaired on failure — so every prediction is concrete enough to be graded.

# PredictionOutput — enforced via tool-use / structured output home_score: int ≥ 0 away_score: int ≥ 0 scorers: [ { player_name: str, team: "home" | "away", likelihood: float # 0 .. 1 } ] match_confidence: float # 0 .. 1 advancing_team: "home"|"away" # knockouts only explanation: str # cites the key signals

🔒 Schema-enforced

The model must satisfy the schema; invalid output triggers a capped repair-retry loop. After N failures the prediction is recorded as failed — never dropped.

🧾 Fully reproducible

Every prediction stores its snapshot, model id, and prompt / schema / calibration version. You can reconstruct exactly why it was made — append-only, never overwritten.

🎯 Stage-aware

Group predictions may be draws; knockout predictions represent the 90-minute scoreline and name the advancing team.

Honest self-scoring

How a prediction is graded

The grading core is a set of pure, deterministic functions — no database, no network, no LLM — so correctness is exhaustively unit-tested. Score-level metrics always use the 90-minute line; own goals are excluded from the scorer set but still count toward total goals.

✅ Exact hit bool

Did the predicted scoreline equal the real 90' scoreline, exactly?

↔ Outcome correct 1 / X / 2

Did we get the direction — home win, draw, or away win — right?

± Goals abs. error int

Absolute difference between predicted and actual total goals.

🎯 Scorer precision & recall 0–1

Of the scorers we named, how many scored — and of those who scored, how many did we name (accent-insensitive matching).

📉 Scorer Brier 0–1

Mean squared error of each named scorer's likelihood vs whether they actually scored.

📊 Confidence Brier 0–1

Squared error of match_confidence against whether the predicted outcome occurred — the calibration signal.

Prediction league

Stage-weighted points

Beyond raw metrics, ScoreSeer keeps a headline "score" — points per match, weighted by how much each stage matters. Scoring is tiered: an exact scoreline earns the higher exact points; a correct outcome alone earns the outcome points; otherwise zero. Exact does not also add the outcome points.

The "learning"

Calibration without a trained model

After each grading run, the system aggregates every graded prediction into observed biases and folds a compact prompt_snippet into future prompts. There is no Poisson or ML model in the MVP — calibration is purely prompt-level, and deliberately isolated so it can be evaluated and swapped later if the signal proves weak.

It needs a minimum of 5 graded matches before it trusts the signal enough to calibrate at all.

⚖ Score bias

Mean over/under-prediction of home & away goals — e.g. "you over-predict favourite blowouts by ~0.4 goals."

🔥 Overconfidence

Mean stated confidence minus realized outcome accuracy — is the system more sure than it should be?

📈 Trend

Confidence Brier in the tournament's second half vs the first — direct evidence the loop is working.

Append-only & auditable

The data model

Postgres, UUID keys, UTC timestamps, JSONB for flexible payloads. This is a correctness & reproducibility problem, not a throughput one — predictions, snapshots, and grades are never mutated; re-runs create new versioned rows.

🗓 Scheduler

APScheduler runs in-process with a persistent job store — max_instances=1, coalescing, plus a Postgres advisory lock so a manual trigger can't race the scheduled run.

🔁 Idempotent

A unique key on (fixture, prompt, model, calibration) makes re-runs a no-op; grades are unique per prediction. Re-running is always safe.

🛟 Resilient

Each fixture runs in its own transaction; one failure writes a failed RunItem and the batch continues — ending "partial," never aborted.

Steering, not guarantees

What "good" looks like

Exact-score prediction is intrinsically low-hit-rate, so the headline signal is outcome accuracy + a calibration trend, not exact-score hits. These are targets to steer by, measured across all graded WC2026 matches.

0
Outcome accuracy
≥ 55% of matches' result direction (1/X/2) predicted correctly by tournament's end.
0
Exact-score hit rate
≥ 12% of exact 90' scorelines correct — vs ~5–8% for blind guessing.
0
Total-goals MAE
Mean absolute error of total goals at or below 1.3.

🎯 Scorer recall ≥ 35%

Of players who actually scored, the fraction the system named — with per-player likelihoods tracked by Brier score.

📐 Calibration improves

Confidence Brier in the second half of the tournament beats the first — evidence the calibration loop earns its keep.

⚙ ≥ 95% operational

At least 95% of scheduled fixtures get a prediction before kickoff and a grade within 24h of full-time.
